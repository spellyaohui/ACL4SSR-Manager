# GitHub 上传通用规则

## 一、目的

本文档用于沉淀一套可跨项目复用的 GitHub 上传规则，统一约束以下动作：

- Git 提交
- Git 推送
- GitHub Release
- 上传前后的文件筛选与收尾检查

本文档强调“通用原则 + 可替换占位符”的方式，适用于大多数软件项目。

不覆盖服务器部署、数据库迁移、服务重启、云资源变更等运行侧规则；如项目另有部署规范，应在项目内补充。

## 二、使用方式

将本文档用于其他项目时，应先结合目标项目补齐以下约定项：

- 默认分支名，例如 `main` 或 `master`
- 版本说明文件路径，例如 `Docs/版本更新说明_YYYY-MM-DD.md`
- 发布脚本命令，例如 `pwsh .\scripts\create-github-release.ps1`
- 前端构建命令，例如 `npm run build`
- 是否要求同步远端服务器或构建产物

如果目标项目没有发布脚本，可以保留本文档的流程约束，但将“执行脚本”替换为该项目实际采用的 Release 步骤。

## 三、通用强制规则

### 1. 优先走统一发布流程

如果项目已经提供统一发布脚本、Make 命令、CI 任务或封装命令，应优先使用，不要长期依赖临时手工步骤。

示例：

```powershell
pwsh .\scripts\create-github-release.ps1
```

或者：

```bash
make release
```

### 2. 提交前先更新版本说明

每次功能、修复、文档更新，在提交前应先创建或更新一份版本说明文档。文件路径由项目自行约定，例如：

```text
Docs/版本更新说明_YYYY-MM-DD.md
```

版本说明建议至少包含：

- 本次完成项
- 接口、页面或模块影响
- 上线后核对建议
- 补充说明

### 3. 影响使用方式时同步更新主文档

若本次改动影响以下任一内容，应同步更新项目主文档，例如 `README.md`、`docs/index.md` 或产品说明文档：

- 功能说明
- 安装或部署步骤
- 数据迁移顺序
- 文档链接
- 定时任务或运行参数

### 4. GitHub 相关文案保持统一语言

同一个项目的以下内容应统一语言风格，避免中英混杂：

- Git 提交信息
- Release 标题
- Release 说明
- 上传到 GitHub 的更新描述

如果项目已明确规定使用中文，则统一使用中文；如果团队约定使用英文，则统一使用英文。

### 5. git add 使用明确白名单

执行 git add 时，只添加本次需要上传的文件，不使用会把无关改动一并纳入的命令。

不推荐：

```powershell
git add .
git add -A
```

推荐：

```powershell
git add src/app.js docs/release-notes/2026-04-05.md
```

### 6. 敏感文件和本地文件不得上传

在任何 git add 之前，必须先排除以下内容：

- 密钥文件
- 凭据文件
- 本地 `.env` 或密文配置
- SSH、数据库、云平台连接信息
- 当前机器专用的临时文件
- 与本次发布无关的工作区残留文件

### 7. 发布依赖缺失时先补环境

如果发布依赖未安装、未登录或未授权，不要直接跳过 Release 步骤，而应先补齐环境。

例如：

```powershell
gh --version
gh auth login
```

## 四、标准上传顺序

每次上传 GitHub，默认按以下顺序执行：

1. 执行 `git status`，确认本次改动范围。
2. 更新版本说明文档。
3. 如有必要，同步更新 README 或其他主文档。
4. 检查并排除敏感文件、本地文件、无关改动。
5. 使用明确文件白名单执行 `git add`。
6. 使用统一语言风格执行 `git commit`。
7. 推送到默认分支或目标分支。
8. 执行项目约定的 Release 流程。

通用示例：

```powershell
git status
git add src/app.js docs/release-notes/2026-04-05.md
git commit -m "fix: 调整登录日志过滤规则"
git push origin main
```

如果项目有统一发布脚本，再执行：

```powershell
pwsh .\scripts\create-github-release.ps1
```

## 五、前端项目的附加规则

如果本次上传包含前端源码变更，发布前应先执行该项目约定的构建命令，确保产物与源码一致。

示例：

```powershell
npm run build
```

要求如下：

- 先生成最新构建产物
- 如果项目需要上传静态资源，应确保远端使用的是最新产物
- 若远端存在旧 hash 文件、重复嵌套目录或历史残留，应在确认新产物可用后再清理

## 六、临时文件处理规则

以下内容默认视为本地临时文件，除非团队明确要求入库，否则不上传到 GitHub：

- 临时测试脚本
- 一次性补数据脚本
- 临时说明文件
- 本地调试目录
- 机器专属缓存文件

对于仅当前机器使用的文件，优先采用以下方式处理：

- 写入 `.git/info/exclude`
- 写入合适的 `.gitignore` 规则
- 任务结束后清理
- 保持 `git status` 尽量干净

## 七、禁止事项

上传 GitHub 时，禁止以下行为：

- 跳过版本说明更新
- 在同一项目中混用不同语言风格的提交和发布文案
- 使用 `git add .` 或 `git add -A` 直接打包全部改动
- 把密钥、凭据、连接信息、`.env` 等敏感文件加入版本控制
- 因发布工具未准备好而直接跳过 Release
- 明明已有稳定发布流程，却长期依赖临时人工命令替代

## 八、发布完成后的检查

每次上传和发布完成后，建议至少记录以下结果：

- 本次 commit hash
- GitHub Release 链接
- 是否已推送到目标分支
- 是否已同步构建产物或部署文件
- 是否还需要人工执行额外步骤

最后再次执行：

```powershell
git status
```

确保工作区没有遗留的未处理文件。

## 九、推荐执行口径

后续凡是“上传 GitHub”“发版”“提交并发布”“推送并出 Release”这类操作，均可默认按本文档执行，再叠加目标项目自己的特定约束。

如果目标项目已有更严格规则，应以项目内规则优先；本文档作为通用基线使用。
